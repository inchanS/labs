Failure
  1. AddKeysToAgent yes 설정을 했지만 재부팅 후 SSH 키가 자동으로 ssh agent에 등록되지 않는다?
  2. ssh key의 암호(passphrase) 설정후 UseKeychain yes 설정을 했는데 왜 자꾸 암호를 물을까?

재부팅 후 SSH 키가 자동으로 ssh agent에 등록되지 않는다?

macOS에서 SSH 클라이언트는 파일 이름이 다음과 같은 경우 기본 키파일명으로서 특별한 처리를 한다.

  • id_rsa
  • id_dsa
  • id_ecdsa
  • id_ed25519
  • 등등…

이러한 기본 키 파일명을 사용할 때 AddKeysToAgent yes와 UseKeychain yes 옵션이 설정되어 있으면, SSH 클라이언트는 자동으로 해당 키를 SSH 에이전트에 추가하고 macOS 키체인에서 암호를 가져온다.

 

보면 알겠지만 기본 키 파일명은 바로 키 생성시 설정한 암호화 알고리즘 옵션명, 즉 id_알고리즘으로 이루어져있다.
따라서 AddKeysToAgent yes 옵션을 사용하고자 한다면 ssh key 파일명은 위에서 언급한대로 설정하여야 한다.

 

그럼에도 불구하고 파일명을 달리하고 싶다면 2가지 방법이 있다.

  1. 각자 다른 암호화 알고리즘으로 키를 생성하여 사용한다.
  2. 재부팅시 자동으로 파일을 등록할 수 있도록 shell 초기화 파일에서 SSH 키를 자동으로 로드하도록 한다.

키체인에서 암호를 가져오지 못하고 매번 묻는다?

키체인에서 암호를 가져오지 못하는 이유는 크게 2가지 상황이 있을 수 있다.

  1. SSH key가 (자동으로) 등록되지 않았을 경우
  2. UseKeychain yes 옵션이 제대로 작동하지 않는 경우

1번의 경우에는 다시 재부팅 후 SSH 키가 자동으로 ssh agent에 등록되지 않는다?를 참고한다.

 

2번의 경우에는 다음의 상황이 있을 수 있다.
해당 레포지토리가 있는 경로에서 git fetch, git push와 같은 명령을 했을 때, 암호를 묻는다. 암호를 묻는다는건 ssh key는 제대로 가져오고 있다는 뜻이다. 그런데 분명 UseKeychain yes 설정을 했는데 왜 계속해서 묻는걸까?

1

ssh-add -l

위 명령으로 현재 ssh agent에 등록된 키를 살펴본다.
이 때 키가 보이지 않을 것이다.

 

이제 아래의 명령으로 ssh 접속 확인을 해본다.

1

ssh -T git@github.com

 

이때 정상적으로 진행이 되어 확인이 된다면 이어서 git fetch 명령을 실행해보자.
아마 암호를 묻지 않고 정상적으로 추가 암호입력 작업 없이 명령이 작동할 것이다.
자 이제 다시 ssh agent에 등록된 키를 살펴본다.
아까는 보이지 않았던 키가 지금은 보일 것이다.

이 이유는 ssh 클라이언트와 에이전트의 작동방식 때문이다.

 

비대화형 환경에서의 키체인 연동 문제:

  • git fetch와 같은 비대화형 명령에서는 UseKeychain yes 옵션이 기대한 대로 동작하지 않을 수 있다.
  • AddKeysToAgent yes 옵션은 SSH 클라이언트가 키를 사용할 때만 동작하며, 자동으로 키를 로드하지 않는다.
  • 키체인은 SSH 클라이언트가 아닌 ssh agent를 통해 암호(passphrase)를 가져온다.
  • ssh -T git@github.com와 같은 명령은 대화형(interactive) 세션에서 동작하며 대화형 세션이 진행될 때, 에이전트에서는 config파일 설정에 따라 에이전트에 다시 키를 로드한다.
  • 때문에 재부팅 후, 바로 ssh 비대화형 명령을 질행하면 SSH 클라이언트가 키체인에서 암호를 가져오지 못하기에 암호를 매번 묻게된다.
  • 따라서 만약 한번이라도 대화형 세션을 진행했다면 키는 정상적으로 ssh agent에 로드되고, 이후 암호를 되묻는 일이 없다.

해결방법

해결 방법은 크게 2가지를 소개한다.

  1. macOS의 launchd를 이용하여 재부팅 및 mac 사용자 로그인시 SSH 키 자동 로드하는 방법
  2. 터미널 최초 실행시 shell 초기화 파일에서 SSH 키 자동 로드

위 2가지 방법 중 하나만 사용하면 되며, 다음 설명 이후 각각의 상황에 맞게 판단하여 사용할 수 있다.

1. macOS의 launchd를 이용하여 SSH 키 자동 로드

macOS의 launchd는 재부팅 및 mac 사용자 로그인시 최초로 한번 작동한다.
즉 재부팅을 하거나 또는 맥 시스템 설정 중 사용자 로그아웃 이후 로그인시 작동한다는 뜻인데, 바로 이 때 SSH 키를 자동로드 할 수 있도록 plist 파일을 하나 추가해주는 방법이다.

1. plist 파일 생성

~/Library/LaunchAgents/ 디렉토리에 com.username.ssh-add.plist 파일을 생성한다. 주의 사항은 위 파일명에서 username 부분을 실제 사용자 이름으로 변경 실제 사용자 이름은 finder에서 홈 디렉토리를 나타낸다.

 

터미널에서 다음 명령어로 확인할 수 있다.

1

whoami

 

이후 나타나는 이름을 그대로 username 부분으로 대치한다.

<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple Computer//DTD PLIST 1.0//EN"
 "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
  <key>Label</key>
  <string>com.username.ssh-add</string>
  <key>ProgramArguments</key>
  <array>
    <string>/usr/bin/ssh-add</string>
    <string>--apple-use-keychain</string>
    <string>/Users/username/.ssh/id_ed25519</string>
  </array>
  <key>RunAtLoad</key>
  <true/>
  <key>StandardErrorPath</key>
  <string>/tmp/ssh-add.err</string>
  <key>StandardOutPath</key>
  <string>/tmp/ssh-add.out</string>
</dict>
</plist>

이 때, 위 코드에서 하이라이트 처리하여 보이는 7번, 12번 line의 ‘username’ 부분을 아까 찾은 홈 디렉토리명으로 동일하게 대치한다.
 

2. launchd 에이전트에 생성한 파일 로드

터미널에서 다음 명령을 실행하여 1. plist 파일 생성에서 생성한 파일을 로드시켜준다.
(역시 username 부분은 홈 디렉토리 명으로 대치해준다.)

1

launchctl load ~/Library/LaunchAgents/com.username.ssh-add.plist

끝, 이제 재부팅하고 테스트해본다.

 

2. shell 초기화 파일에서 SSH 키 자동 로드

이 방법은 최초 터미널 실행시 로드되는 쉘 초기화 파일에서 SSH 키를 자동으로 로드시키는 방법이다.
~/.bash_profile 또는 ~/.zshrc 파일에 다음을 추가한다.

1

ssh-add --apple-use-keychain ~/.ssh/id_ed25519 &>/dev/null

재부팅 후, 터미널 실행시 해당 명령이 실행되면서 이제 비대화형 세션(git fetch와 같은)을 처음 진행하더라도 암호를 되묻지 않는다.

이 방법은 SSH key 생성시 기본 파일명이 아니더라도 AddKeysToAgent yes 옵션이 마치 항상 실행하듯 해준다는 장점이 있다.

Warning
이 방법은 터미널을 열 때만 실행되므로, GUI 환경이나 터미널을 열지 않고 git 명령을 실행할 때는 키가 로드되지 않는다.
